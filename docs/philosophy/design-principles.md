# NEST Design Principles — 物としてどうあるか

> 🧭 **ここは思想層です。** この文書はすべての成果物を設計するときの基準となる、NESTの**定規（Design Principles）**です。

- **Version**: 1.0
- **発行**: AT-Labo

---

**位置づけ**: NESTから生まれるすべての成果物 — Portal・Handbook・Apps・Knowledge・Journey・Starter Kit、そしてまだ名前のないもの — を設計するときの判断基準。[Brand Book](brand-book.md)が「人としてどうあるか」なら、これは「**プロダクトとしてどうあるか**」を定める定規である。

**適用範囲**: HTMLでも、アプリでも、ドキュメントでも、フォルダ構成でも適用できる。「画面」の原則ではなく「体験」の原則だからである。

## 原則1. ひとつのことを、ちゃんとやる

すべての成果物は、担当をひとつだけ持つ。ふたつの仕事をしているものは、ふたつに分ける。

- **こうする**: ページなら1ページ1目的。アプリなら「一言で何をするもの」と言い切れる。ドキュメントなら冒頭1行で役割を宣言する
- **こうしない**: 「ついでにこの機能も」「せっかくだからこの説明も」で膨らませない
- **検品の問い**: *「これは何をするもの？」に一文で答えられるか？*

## 原則2. 最初の「できた！」までを、最短にする

理解してから使うのではない。使えたから理解したくなる。すべての成果物は、触れた人が最初の成功体験に到達するまでの時間で測る。

- **こうする**: 開いて3分以内に「何かができた」を用意する。完璧な説明より、動く最小の一歩
- **こうしない**: 前提知識の説明を成功体験より先に積み上げない。「まずこの章を全部読んでから」と言わない
- **検品の問い**: *初めての人は、何分で最初の「できた！」に着くか？*

## 原則3. 壊しても、戻れるようにする

安心は、注意書きではなく**構造**で作る。取り返しがつくと知っているから、人は挑戦できる。

- **こうする**: すべての変更は記録され、いつでも戻せる状態を保つ。危険な操作の前には確認を置く。「戻り方」を「進み方」とセットで書く
- **こうしない**: 「注意してください」で済ませない（注意で防げるなら苦労しない）。戻れない操作を初心者の通り道に置かない
- **検品の問い**: *最悪の操作ミスをしたとき、この人は無事か？*

## 原則4. 隠さない。ただし、いっぺんに見せない

情報に鍵をかけない。しかし全部を一度に浴びせない。**深さは段階で、扉は全開で。**

- **こうする**: 最初は要点だけ、知りたい人には深い階層を用意する（概要→詳細→根拠）。どの階層も誰でも読める
- **こうしない**: レベルや権限で情報をロックしない。逆に、初対面の人に全ルールを一枚で見せない
- **検品の問い**: *初心者は圧倒されず、探求者は物足りなくないか？*

## 原則5. 「いまどこ？」に、いつでも答える

迷子は、能力の問題ではなく設計の失敗である。すべての成果物は、現在地と次の一歩を常に示す。

- **こうする**: 冒頭に「ここは何の場所か」、末尾に「次はどこへ」。作業には現在地メモ（STATUS）、成長には地図（[Journey](journey.md)）
- **こうしない**: リンクの行き止まりを作らない。「どこから来てどこへ行くのか」不明なページを放置しない
- **検品の問い**: *ここに突然着地した人は、自分の位置がわかるか？*

## 原則6. 流行より、標準を選ぶ

10年後も読める形式、どの道具でも開ける形式を選ぶ。特定の製品・サービス・AIが消えても、NESTは無傷でいる。

- **こうする**: プレーンテキストとMarkdownを土台にする。開かれた標準・枯れた技術を優先する。便利なサービスは「窓口」として使い、本体はいつも自分の手元に置く
- **こうしない**: 特定ツールでしか開けない形式に大事なものを入れない。流行しているという理由だけで採用しない
- **検品の問い**: *この道具が明日消えたら、何を失う？（答えが「何も」なら合格）*

## 原則7. 完成させず、育つ余白を残す

Version 1.0は完成ではなく誕生。将来の成長を**先回りして作り込まず**、成長できる**余白だけ**を設計する。

- **こうする**: 今必要な最小を作り、拡張の「置き場所」だけ決めておく。機能追加は「本当に困ってから」
- **こうしない**: 「いつか使うかも」で作らない。10個になったら必要になる仕組みを、2個の時点で作らない
- **検品の問い**: *これは「いま困っていること」を解決しているか？　それとも想像上の未来か？*

## 原則同士がぶつかったら

1. **Brand Book（憲法）が常に最優先**（例: 原則7「余白」より六箇条「初心者を置いていかない」が勝つ — 初心者への説明は「余白」の名目で削らない）
2. 原則同士の衝突は、**初心者にとってどちらが優しいか**で決める
3. それでも決まらなければ、決めた理由を記録して選ぶ（間違っていたら戻ればいい — 原則3を自分たちにも適用する）

## 参考にした思想への感謝（受け取った恩恵）

| 源流 | 受け取ったもの |
|---|---|
| Apple Human Interface Guidelines | 寛容さ（取り返しのつく操作）・段階的開示 → 原則3・4 |
| Google Material Design | 一貫した比喩が学習を不要にするという思想 → 原則1・5 |
| Basecamp | 小さく作る勇気・「機能を足さない」という決断 → 原則7 |
| GitHub | プレーンテキストとバージョン管理が最強の保存形式 → 原則3・6 |
| Notion | 情報に階層を、しかし壁は作らない → 原則4 |
