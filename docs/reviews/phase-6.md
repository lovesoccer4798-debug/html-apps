# Phase 6 レビュー記録 — デザインシステム（Design Tokens）

- **日付**: 2026-07-08
- **担当**: Opus 4.8（モデル切替後・思想は継続）
- **成果物**: `apps/portal/tokens.css`・`templates/app/tokens.css`、`style.css`のトークン消費化、Dashboardの作業面色、`docs/design-guide.md`（トークン体系・役割差別化・非肥大ガバナンス）、[ADR](../adr/20260708-design-tokens.md)

## 目的

見た目の値を一箇所（tokens.css）に集約し、10年運用と「新しい巣が同じ見た目で育つ」土台を作る。あわせてオーナーの継続レビュー4点（役割のデザイン差別化・Dashboard非肥大・Design Tokens・Portal検索）を処理する。

## レビュー結果

- 色・影・角丸が style.css に直値で散在し、ダーク対応もコンポーネント側にあった → トークンへ集約
- Phase順: 「最初の住人」はオーナー参加＋Pages有効化が前提でブロック中。待ち時間にデザインシステムを先行させる方が、オーナーの最初のアプリが整ったtokensを継承でき合理的と判断
- スクリーンショットで「見た目は現状維持（回帰なし）」＋「Dashboardが作業面色で微差別化」を確認

## 採用

- **Design Tokens 分離**（ADR参照）: 値=tokens.css／コンポーネント=style.css／方針=design-guide.md
- **役割のデザイン差別化**: 同一トークン上でDashboardのみ `--nest-bg-alt`。色と密度で「入口」と「作業場」を伝える（ブランドは割らない）
- **Dashboard非肥大ガバナンス**: 6グループ固定・1グループ最大5リンク・新情報はHandbook/Knowledgeへ・グループを足さない（design-guideに明文化）
- **Portal全体検索**: Future候補としてroadmapに記載（今は作らない・原則7）

## 却下

- ルート共有の単一トークンファイル（アプリ単独切り出しで壊れる。extractability優先）
- CSSフレームワーク導入（外部依存・ロックイン）
- 余白/文字スケールのトークン化（現時点で消費先がない・原則7。使う時に追加）

## 理由

見た目の一貫性は「値の正本を1つにする」ことで構造的に保証できる。手書きで各アプリの色を揃える運用は必ず崩れる（鉄則2と同じ発想）。Portal/Dashboardの差別化を新しい色や装飾でなく既存トークンの differ で表現したのは、ブランドの一貫性（1つのアクセント）を守るため。
