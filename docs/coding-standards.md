# コーディング規約

言語・フレームワークを問わず適用する共通原則と、言語別の規約を定める。新しい技術スタックを採用したら、このファイルに言語別セクションを追記して育てる。

## 共通原則（すべての言語に適用）

1. **読みやすさ > 賢さ。** トリッキーな1行より、素直な3行。
2. **既存コードに合わせる。** ファイルを編集するときは、そのファイルの命名・スタイル・イディオムを踏襲する。個人の好みを持ち込まない。
3. **命名は英語で、意味が伝わる長さで。** 略語は一般的なもの（`id`, `url`, `max`）のみ。
4. **コメントは「なぜ」を書く。** コードを読めばわかる「何を」のコメントは書かない。制約・意図・落とし穴のみ記す。
5. **関数は小さく、1つの責務。** 目安として1関数50行を超えたら分割を検討する。
6. **マジックナンバーを避ける。** 意味のある定数名を付ける。
7. **エラーを握りつぶさない。** 例外は捕捉したら、処理するか、ログを残して再送出する。
8. **秘密情報をコードに書かない。** 環境変数または gitignore された設定ファイルを使う。
9. **依存は最小限に。** ライブラリ追加は「自分で書くより明確に得か」を判断し、採用したらADRに理由を残す。

## フォーマットとリンター

- フォーマットは機械に任せる。手で整えない。
- スタック採用時にフォーマッター/リンターを決めて設定ファイルをリポジトリにコミットし、下の言語別セクションに記載する。

## 言語別規約

### HTML / CSS / JavaScript（ブラウザ向け小規模アプリ）

- 1ファイル完結の小さなアプリは、`apps/<アプリ名>/index.html` にHTML/CSS/JSをまとめてよい。規模が育ったらファイル分割する。
- インデントはスペース2つ。
- JavaScriptは `const`/`let` を使い、`var` を使わない。
- DOM操作は `querySelector` 系に統一。
- 外部CDNへの依存は最小限にし、使う場合はバージョンを固定する。

### （新しいスタックを採用したらここに追記）

例: TypeScript / Python / Swift / Kotlin など。追記の際は以下を含めること。

- フォーマッター・リンターと設定ファイルの場所
- ディレクトリ構成の慣習
- テストフレームワークと実行コマンド
