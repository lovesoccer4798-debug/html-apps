# 開発フロー

このワークスペースでの開発は、常に次の流れに従う。どのAIツールを使っていても同じフローを適用する。

```
アプリ作成 → 仕様を書く → タスク化 → 実装 → セルフレビュー → テスト → コミット/PR → STATUS更新
(templates/app) (apps/*/specs) (tasks/)  (apps/)  (review-process)  (testing-policy) (git-workflow) (tasks/STATUS.md)
```

## 0. 新しいアプリを始める

```bash
cp -r templates/app apps/<アプリ名>
```

- アプリ名は英語ケバブケース（例: `todo-app`）。
- コピーした `README.md` の概要・起動方法を埋め、`tasks/STATUS.md` のプロジェクト索引に1行追加する。
- アプリ局所の成果物（仕様書・CHANGELOG）は必ず `apps/<アプリ名>/` 配下に置く。ルートに置かない。

## はじめてモード（正式な例外規定）

**Journey Level 3の「最初の1個」に限り、仕様書とタスク管理を免除する。** `prompts/first-app.md` を使い、AIと一緒に作る楽しさを最優先する（HTMLファイル1個・小さな完成を目指す）。**2個目のアプリから本フローを適用する** — 正式フローは制約ではなく、育てたいものができた人への次のステップとして案内すること。

## 1. 仕様を書く（apps/<アプリ名>/specs/）

- 新しい機能を作るときは、**コードを書く前に** `templates/spec.md` をコピーして `apps/<アプリ名>/specs/<機能名>.md` を作る。
- 小さな変更（バグ修正、微調整）は仕様書不要。既存仕様に影響するなら該当仕様書を更新する。
- 仕様書には「何を作るか」「なぜ作るか」「完成の定義」を必ず書く。実装方法は書かなくてよい。

## 2. タスク化（tasks/）

- 仕様をもとに、`templates/task.md` をコピーして `tasks/backlog/` にタスクファイルを作る。ファイル名にアプリ名を含める（例: `20260707-todo-app-mvp.md`）。
- 1タスクは「1セッションで完了できる粒度」を目安にする。大きければ分割する。
- 着手時に `tasks/in-progress/` へ移動する。in-progress は**1人（1エージェント）につき1つ**まで。

## 3. 実装

- `AGENTS.md` と `docs/coding-standards.md` に従う。
- 重要な設計判断（ライブラリ選定・データ構造・アーキテクチャ）をしたら `docs/adr/` に記録する。
- 実装中に仕様の曖昧さや矛盾を見つけたら、仕様書を直してから実装を続ける。仕様と実装を乖離させない。

## 4. セルフレビュー

- `docs/review-process.md` のチェックリストに従い、コミット前に自分（AI自身を含む）でレビューする。

## 5. テスト

- `docs/testing-policy.md` に従う。最低限、変更した箇所が実際に動くことを確認してからコミットする。

## 6. コミット / PR

- `docs/git-workflow.md` に従う。作業ブランチ → コミット → push → 必要ならPR。
- ユーザーに見える変更は、**該当アプリの** `CHANGELOG.md` の `[Unreleased]` セクションに追記する（ワークスペース基盤の変更はルートの `CHANGELOG.md`）。

## 7. 作業の締め（毎セッション必須）

セッションを終える前に、必ず以下を行う。**これが「30分で再開できる」状態を保つ鍵。**

1. `tasks/STATUS.md` を更新する（完了したこと・残っていること・次の一手・ハマりどころ）
2. 完了したタスクを `tasks/done/` へ移動する
3. 未コミットの変更を残さない（WIPでもコミットしてpushする）

## 途中で中断するとき

作業が中途半端でも、上記「作業の締め」を実行する。中断状態の詳細（どのファイルのどこまで書いたか、何を試して何がダメだったか）はタスクファイルの「作業ログ」欄に書く。

## アプリを単独リポジトリへ切り出すとき

アプリが育って独立させたくなったら、`apps/<アプリ名>/` をそのまま新リポジトリのルートにコピーすればよい（仕様書・CHANGELOGが同梱されているため情報が欠けない）。切り出し後、このリポジトリ側では該当ディレクトリを削除し、`tasks/STATUS.md` の索引に移転先リンクを残す。
