# テスト方針

## 基本姿勢

- **「動くことを確認せずにコミットしない」が最低ライン。** 自動テストがないプロジェクトでも、変更した機能を実際に動かして確認する。
- テストは「未来の自分と別のAIが安心して変更するための保険」。AIを切り替えたとき、テストが通ることが品質の共通言語になる。
- テストコードもプロダクションコードと同じ品質基準で書く。

## テストの優先順位

リソースは有限なので、次の順で投資する。

1. **コアロジックのユニットテスト** — 計算・変換・判定など、壊れると致命的な純粋ロジック
2. **ハッピーパスの結合テスト** — 主要ユースケースが最初から最後まで通ること
3. **バグ再発防止テスト** — バグを直したら、必ずその再現テストを追加する（回帰防止）
4. **エッジケース** — 空入力・境界値・異常系

UIの見た目や些末なスタイルの自動テストは原則書かない（費用対効果が低い）。

## プロジェクト種別ごとの方針

| 種別 | 方針 |
|---|---|
| 小規模HTMLアプリ | 手動確認でよい。確認手順を仕様書の「完成の定義」に書く |
| 業務自動化スクリプト | 入出力のユニットテスト必須。実データを壊さないようドライランモードを用意する |
| Web/モバイルアプリ | コアロジックにユニットテスト、主要フローに結合テスト |
| AIエージェント | LLM出力は非決定的なので、ツール呼び出し・パース・状態管理などの決定的な部分をテストする。LLM部分は評価用の入出力例を `specs/` に記録する |

## 実行ルール

- **push前にテストを実行する。** 失敗したままpushしない。
- テストの実行コマンドは各プロジェクトの README に明記する（例: `npm test`）。コマンドがドキュメントにない状態を作らない。
- 失敗したテストを「とりあえずスキップ」で放置しない。スキップするなら理由をコメントに書き、タスク化する。

## テストが書きにくいと感じたら

テストしづらいコードは設計が悪いサイン。副作用（I/O・DOM・API呼び出し）とロジックを分離できないか検討する。
