# ADR-0002: AI共通ルールをAGENTS.mdに一元化し、ベンダー固有ファイルはスタブとする

- **日付**: 2026-07-06
- **ステータス**: 採用
- **関連**: `AGENTS.md`, `CLAUDE.md`, `GEMINI.md`, ADR-0001

## 背景（Context）

AIコーディングツールはそれぞれ独自のルールファイルを読む（Claude Code→`CLAUDE.md`、Gemini CLI→`GEMINI.md`、Cursor→`.cursor/rules` など）。ツールごとにルールを書き分けると内容が乖離し、AIを切り替えたときに品質・ルールが変わってしまう。

## 検討した選択肢

### 選択肢A: `AGENTS.md` を唯一のルールファイルとし、各ツール用ファイルは参照スタブにする

- 長所: ルールが一箇所に集約され乖離しない。`AGENTS.md` は Codex / Cursor / Gemini CLI 等が対応する事実上の中立標準
- 短所: スタブを読まず `AGENTS.md` に辿り着かないツールが存在しうる（その場合はスタブに参照を書くことで誘導できる）

### 選択肢B: ツールごとにルールファイルを複製する

- 長所: 各ツールのネイティブな読み込みに最適化できる
- 短所: 複製の同期コストが高く、必ず乖離する。ベンダーロックインの温床

### 選択肢C: シンボリックリンクで各ツール用ファイル名を `AGENTS.md` に向ける

- 長所: 内容の乖離が構造的に起きない
- 短所: Windows環境やGit設定によってシンボリックリンクが正しく機能しないリスク

## 決定（Decision）

選択肢Aを採用する。ルール本体は `AGENTS.md` のみに書き、`CLAUDE.md`・`GEMINI.md` は「AGENTS.mdを読め」とだけ書いたスタブとして維持する。新しいAIツールを使い始めて専用ファイルが必要になったら、同形式のスタブを追加する。

## 影響（Consequences）

- 良くなること: どのAIでも同じルールが適用される。ルール変更が1ファイルの編集で完結する
- 引き受けるリスク: スタブの自動追従はないため、新ツール導入時にスタブ追加を忘れる可能性がある（`README.md` の構成図に記載して防ぐ）
- 覆す条件: 主要ツールがすべて `AGENTS.md` をネイティブサポートした場合、スタブ自体を削除できる
