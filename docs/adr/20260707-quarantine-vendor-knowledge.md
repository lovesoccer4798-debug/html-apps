# ADR: ベンダー固有のセットアップ知識をdocs/ai-tools.mdに隔離する

- **日付**: 2026-07-07
- **ステータス**: 採用
- **関連**: `docs/ai-tools.md`, [AGENTS.md一元化](20260706-ai-agnostic-workspace.md)

## 背景（Context）

ルール本体はAGENTS.mdに一元化したが、「各AIツールにどうAGENTS.mdを読ませるか」（スタブの要否、設定ファイルの書き方、ツール固有の挙動）というベンダー固有の知識はゼロにできない。この知識が個人の頭の中や複数ドキュメントに散らばると、AI乗り換え時の障害になる。

## 検討した選択肢

### 選択肢A: `docs/ai-tools.md` を新設し、ベンダー知識をこの1ファイルにのみ書く（検疫方式）

- 長所: 変更箇所が常に「スタブ1個＋表1行」で済む。乗り換え時に読む場所が1つ。他のドキュメントは中立を保てる
- 短所: 新ツール導入時に表の更新を忘れる可能性がある（→ 導入手順自体を同ファイルに明記して防ぐ）

### 選択肢B: ベンダー知識を書かない（各自がツールのドキュメントを読む）

- 長所: ドキュメントが1つ減る
- 短所: 「このワークスペースでの」設定方法（スタブ方式・確認方法）はツール公式ドキュメントには載っていない。乗り換えのたびに再調査が発生し「30分で再開」を損なう

## 決定（Decision）

選択肢Aを採用する。あわせてWindsurf用スタブ（`.windsurf/rules/workspace.md`）を追加し、主要5ツール（Claude Code / Codex / Cursor / Gemini CLI / Windsurf）の対応状況表を整備した。

## 影響（Consequences）

- 良くなること: AI乗り換えの手順が「ai-tools.mdを見る」に定型化される
- 引き受けるリスク: 表の「未検証」項目は実際に使うまで動作保証がない（使ったら検証結果を反映する運用）
- 覆す条件: 全ツールがAGENTS.mdをネイティブサポートし、スタブも設定も不要になったら、このファイルは対応状況表だけに縮小できる
