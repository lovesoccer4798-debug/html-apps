# ADR: 最小CIでルール遵守を機械的に保証する

- **日付**: 2026-07-07
- **ステータス**: 採用
- **関連**: `.github/workflows/ci.yml`, `AGENTS.md`

## 背景（Context）

このワークスペースのルール（秘密情報をコミットしない、ドキュメントのリンクを壊さない等）は、当初すべて文章で書かれているだけで、遵守はAIの読解力と善意に依存していた。AIツールを乗り換えたとき品質が変わる最大の要因は、この「ルール解釈・遵守度の差」である。ベンダー非依存を掲げる以上、品質の下限は機械が保証する必要がある。

## 検討した選択肢

### 選択肢A: GitHub Actionsで最小CI（gitleaks＋オフラインリンクチェックの2ジョブのみ）

- 長所: どのAIが作業しても同じゲートを通る。無料枠内（Markdown中心なら数十秒/回）。設定はYAML2ファイル分の知識で済む
- 短所: GitHub Actionsへの軽い依存（ただし処理自体は標準的なOSSツールで、他CIへの移植は容易）

### 選択肢B: pre-commitフック（ローカル実行）

- 長所: プッシュ前に検出できる
- 短所: 各開発者・各AI実行環境でのセットアップが必要で、セットアップ漏れをすり抜ける。強制力がない

### 選択肢C: 導入しない（ドキュメントのみ）

- 長所: 依存ゼロ
- 短所: 現状の弱点（善意頼み）が放置される

## 決定（Decision）

選択肢Aを採用する。**意図的に2ジョブに限定**する（CHANGELOG更新チェック等の凝った自作スクリプトは保守負債になるため見送り）。ジョブを追加する場合は新しいADRを残す。Bは将来Aの補完として任意導入してよい。

## 影響（Consequences）

- 良くなること: 秘密情報の混入とドキュメントのリンク切れが、AIの注意力と無関係に検出される
- 引き受けるリスク: GitHub以外のホスティングに移る場合はCI設定の書き換えが必要（ツール自体はどこでも動く）
- 覆す条件: CIが頻繁に誤検知して開発を妨げるようになったら、ルールの調整または縮小を検討する
