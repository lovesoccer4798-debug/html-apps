# ADR: デザインの値を tokens.css に集約し、Design Systemの土台とする

- **日付**: 2026-07-08
- **ステータス**: 採用
- **関連**: `apps/portal/tokens.css`, `templates/app/tokens.css`, `docs/design-guide.md`, [アイコン方式](20260708-icon-system.md), [アプリ同梱構造](20260707-colocate-app-artifacts.md)

## 背景（Context）

Portal・Dashboardの見た目の値（色・影・角丸）が `style.css` の中に直接埋め込まれ、ダーク対応もコンポーネント側に散っていた。今後アプリが増えると、各アプリがCSSをまるごとコピーし、色がバラつき、一括変更もできなくなる。10年運用と「新しい巣が同じ見た目で育つ」（Starter Kit＝DNA）のために、見た目の値を一箇所に集約する必要があった。あわせてPhase順の改訂（「最初の住人」はオーナー参加待ちのため「デザインシステム」を先行）も記録する。

## 検討した選択肢

### 選択肢A: `tokens.css`（値）と `style.css`（コンポーネント）を分離し、各アプリがtokens.cssのコピーを持つ

- 長所: 値の正本が1ファイル。ライト/ダークもトークン側で完結。新アプリはtokens.cssをコピーするだけで見た目を継承（DNA）。アプリ同梱構造・単独切り出しと両立
- 短所: アプリ間でtokens.cssが複製される（DRYではない）

### 選択肢B: ルート共有の1ファイルを全アプリが参照

- 長所: 複製ゼロ・完全なDRY
- 短所: アプリが共有ファイルに依存し、単独リポジトリへの切り出しで壊れる（アプリ同梱構造・extractabilityに反する）

### 選択肢C: CSSフレームワーク/デザインシステムライブラリを導入

- 短所: 外部依存・ビルド・ベンダーロックイン。JSゼロ・依存ゼロ方針に反する。却下

## 決定（Decision）

選択肢Aを採用。値＝`tokens.css`、コンポーネント＝`style.css`、方針＝`design-guide.md` に三分割する。複製は「DNAは各細胞が同じものを持つ」考え方で受容（アプリ同梱構造ADRと同じ判断軸＝extractability優先）。余白・文字スケールのトークン化は使う必要が出るまで作らない（原則7）。Portal/Dashboardは同一トークン上で背景色（`--nest-bg` / `--nest-bg-alt`）だけ変え、役割をデザインで自然に伝える。

## 影響（Consequences）

- 良くなること: 色の一括変更が1ファイルで完結。新アプリの見た目が自動で揃う。ダーク対応の二重管理が消える
- 引き受けるリスク: tokens.cssがアプリ間で複製される（値変更時は各コピーへ伝播が必要。design-guideに伝播ルールを明記）
- 覆す条件: アプリが多数になり複製の伝播が負担になったら、Bへの移行（共有＋切り出し時コピー）を再検討する
