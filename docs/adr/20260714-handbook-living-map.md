# ADR: Handbookを「事実を持たない生きた地図」として改組する

- **日付**: 2026-07-14
- **ステータス**: 採用
- **関連**: [roadmap Phase 9](../roadmap.md)／[タスク](../../tasks/done/20260714-handbook.md)／[ADR: design-tokens](20260708-design-tokens.md)／[ADR: icon-system](20260708-icon-system.md)

## 背景（Context）

Workspace全体をAIは理解できるが、人間が迷わず使える状態になっていなかった。人間向けの「唯一のSingle Source of Truth＝生きた地図」としてHandbookを設計する必要があった。制約は、①ロードマップの3つの鉄則（正本はMarkdownだけ・手書き複製ゼロ・動的情報は自前実装しない）、②AIのトークンコストを増やさない、③NotebookLMの知識ベースを兼ねる、④予算制約のため新規成果物は最小に。

また設計中に、roadmapのPhase 7状態列が更新漏れ（STATUSは完了・roadmapは⬜）である実例が見つかり、「同じ事実を2箇所に書くと必ず片方が古びる」ことが裏付けられた。

## 検討した選択肢

### 選択肢A: 新しいHTMLアプリとしてHandbookを新築

- 長所: 触れる体験のリッチさ
- 短所: 鉄則1違反（正本がMarkdownでなくなる）。既存のHandbook表紙・Dashboard・STATUSと役割が衝突し、更新箇所が増える

### 選択肢B: 既存 `docs/README.md`（Handbook表紙）を改組＋最小の増築（採用）

- 長所: パス不変で全既存リンクが生きる。docs/配下なのでNotebookLMへ自動同期。新規ファイルは1枚だけ
- 短所: Markdownの表現力の範囲に縛られる（→ 触れる体験はCreator Studio等への2クリック導線で実現）

## 決定（Decision）

選択肢Bを採用し、以下の設計判断をセットで決めた。

1. **Handbookは事実を所有しない（1情報1正本）。** 表紙は「地図・ポインタ・動線」だけを持ち、事実（進捗・状態・数字）は既存の正本（STATUS・roadmap・Releases）に置いたまま指す。古びても「嘘」ではなく「案内の遅れ」で済む構造にする
2. **例外は「鮮度情報」だけ。** 「この地図の最終更新」「NotebookLM同期日」の正本を表紙の品質パネルに置く（NotebookLM同期日はSTATUS申し送りから移設）。鮮度の正本は鮮度を表示する場所に置くのが最も腐りにくい
3. **Todayを最上位に置く。** Handbookは説明書ではなく毎日開くホーム。現在地1行（Phase完了時にのみ変わる低頻度事実）＋STATUSへの太い導線。「次の一歩」の本文はSTATUSが正本のため転記しない
4. **目的導線（やりたいことから探す）を第2ブロックに。** ラベルは「名前」ではなく「動詞＋所要時間」。すべての目的は2クリック以内で「触れるもの」に着地する
5. **3ホームの役割を再定義。** Portal＝玄関（迎える）／Handbook＝毎日開くホーム（現在地と目的の入口）／Dashboard＝近道（全部へ2クリックのリンク集）。語彙台帳を正本として各画面の文言を揃える
6. **図はテキスト図（コードブロック）で描く。** Mermaidは GitHub Pages（minima）で描画されずコード表示になり、初見の理解を壊すため。テキスト図はGitHub・Pages・NotebookLMのどこでも同じに読める。各図の直下に1〜2行の文章要約を必ず添える（NotebookLM対応・アクセシビリティ）
7. **Markdown層では装飾絵文字を使わない。** 語彙台帳が定める場所の識別子のみ最小限に使う（デザインガイドの二層方針に準拠。UIアイコンはLucide）
8. **Handbookの非肥大ガバナンス。** Handbookは表紙＋ `docs/handbook/` 2枚（atlas）まで。増やしたくなったら既存の章（development-flow等）側に書く
9. **AIの必読リストは変えない。** AGENTS.mdの「作業開始時に読むもの」にHandbookを加えない（トークンコスト分離）。Handbookは人間向け、AGENTS.mdはAI向け
10. **scenes.md（シーン別動線）は作らない。** 表紙の目的導線が役割を吸収（引き算）
11. **Navigatorは「育成中のAI運用思想」として扱う。** 実体アプリが存在する前提で設計を広げない。当面その役割はAGENTS.mdが担っていることを地図に明記する

## 影響（Consequences）

- 良くなること: 人間の入口が1つに定まる。更新対象が「現在地1行＋品質パネル」に局所化され、Phase完了チェックと月1点検で守れる。NotebookLM同期は既存フローのまま自動対象
- 悪くなること・引き受けるリスク: 現在地1行と品質パネルは手動更新であり、更新を忘れると古びる（検知は月1のNotebookLM検収質問とセルフレビューのチェック項目で行う）。テキスト図はMermaidより見た目が素朴
- 将来この決定を覆す場合の条件・コスト: PagesでMermaid描画が必要になったら（Future候補のTrigger参照）図の形式だけ差し替え可能。Handbookが2枚で足りなくなったら、それは章側の整理の合図
