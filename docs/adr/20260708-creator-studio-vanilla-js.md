# ADR: Creator Studio は Vanilla JavaScript（依存ゼロ）で作る

- **日付**: 2026-07-08
- **ステータス**: 採用
- **関連**: `apps/creator-studio/`, `docs/design-guide.md`, [Jekyll採用](20260707-jekyll-pages.md), [Design Tokens](20260708-design-tokens.md)

## 背景（Context）

NEST最初の実用アプリ Creator Studio は、Portal（初のJSなしアプリ）と違い、入力→プロンプト生成→コピー→設定保存という**対話的な動作**が必要で、JavaScriptが要る。そこで「どのJavaScript環境で作るか」を決める必要があった。選択は、このアプリだけでなく **NESTで今後作るアプリ全体の既定** になる（Starter KitのDNAにも影響する）ため、ADRとして残す。

前提となるNESTの制約・思想:
- Design Principles 原則6「流行より標準を選ぶ」・「便利なサービスは窓口として使い、本体は手元に置く」
- 10年運用・ベンダーロックイン回避・無料運用
- 配信はGitHub Pages（静的・ビルドなし）。`_config.yml` 1枚でMarkdownをHTML化している（[Jekyll ADR](20260707-jekyll-pages.md)）

## 検討した選択肢

### 選択肢A: Vanilla JavaScript（フレームワークなし・ビルドなし・CDNなし）

- 長所:
  - **依存ゼロ**。`index.html` をブラウザで開くだけで動く。ビルド工程・node_modules・パッケージ更新が存在しない
  - 10年後もそのまま動く可能性が最も高い（ブラウザの後方互換は極めて強い。フレームワークのメジャー更新・EOLに振り回されない）
  - GitHub Pagesの静的配信・Portalの「JSゼロ」路線と一貫。学習対象が「素のWeb標準」だけで、初心者にも読める
  - 今回の規模（1画面・数百行）では十分。データ駆動（MEDIA/SETTINGS）＋純粋関数（PromptEngine）で保守性は確保できている
- 短所:
  - 状態が増えた時、手書きのDOM更新が煩雑になりうる（大規模UIでは不利）
  - コンポーネントの再利用の仕組みは自前

### 選択肢B: React / Vue（コンポーネントフレームワーク）

- 長所: 大規模UI・複雑な状態管理に強い。エコシステムが広い
- 短所:
  - **ビルド工程（Vite等）・node_modules・依存更新**が発生。GitHub Pagesの「置くだけ配信」から外れ、CI/ビルドという保守対象が10年増え続ける
  - フレームワークのメジャー更新・破壊的変更・EOLに追従コストがかかる（10年でReactは何度も変わる）
  - 1画面のアプリには過剰。ベンダー（ライブラリ）ロックインの入り口
  - CDN読み込み版もあるが、外部CDN依存はNESTの「CDNを使わない」方針に反する

### 選択肢C: Alpine.js（軽量・HTML属性ベース・CDN 1本）

- 長所: 学習が容易、少量のインタラクションに手軽。ビルド不要
- 短所:
  - 標準的には**CDNから読み込む**運用が前提で、NESTの「CDNを使わない・本体は手元に置く」に反する（自前ホストは可能だが、それはライブラリ1つを抱える判断）
  - フレームワーク独自のディレクティブ（`x-data` 等）を学ぶ必要があり、「素のWeb標準だけ」より学習対象が増える
  - 依存が1つ増える。今回はそれに見合う複雑さがない

## 決定（Decision）

**選択肢A（Vanilla JavaScript・依存ゼロ）を採用する。** あわせて、これを **NESTアプリの既定** とする。

依存ゼロを維持する理由:
1. **10年運用**: 依存が増えるほど、更新・脆弱性対応・EOL追従という「持ち主が変わっても続く負債」が積み上がる。ゼロなら腐らない
2. **ベンダーロックイン回避**: フレームワークもまた乗り換えコストを生む依存先。NESTの中核思想（特定技術に縛られない）と一致
3. **初心者最優先**: 学習対象が「素のHTML/CSS/JS」に閉じる。別のAIや未来の自分が最小の前提知識で読める
4. **無料・静的配信**: ビルドもサーバも要らず、GitHub Pagesにそのまま載る

品質はフレームワークの代わりに**設計**で担保する: データ駆動（MEDIA/SETTINGS配列）、責務分離（DATA / Store / PromptEngine / UI の一方向依存）、純粋関数（PromptEngine.build）。

## 将来フレームワーク導入を検討する条件（＝この決定を見直すトリガー）

次の**いずれかが実際に起きたら**、フレームワーク導入を検討する（それまでは導入しない＝原則7）:

- 1つのアプリで**画面・状態が大きく増え**、素のDOM更新の手書きが**バグの温床**になったとき
- 複数画面・ルーティング・複雑なフォーム状態など、**フレームワークが明確に工数を減らす**規模になったとき
- 実運用で**パフォーマンスやアクセシビリティ**に、自前実装では割に合わない課題が出たとき

導入する場合でも: (1) ビルドなし or 最小で動くこと、(2) 外部CDN常時依存にしないこと（ローカル同梱）、(3) そのアプリ単独に閉じ、他アプリ・NEST基盤へ強制しないこと、を条件とし、判断は新しいADRに残す。

## 影響（Consequences）

- 良くなること: アプリが「置くだけで動く」状態を保てる。保守対象（依存・ビルド）が増えない。Starter Kitで配っても相手の環境を選ばない
- 引き受けるリスク: 将来、大規模UIを素のJSで作り続けると開発効率が落ちうる → 上記トリガーで個別に見直す
- 覆す条件: 上記トリガーのいずれかが実証されたとき、そのアプリ単位で（NEST全体ではなく）フレームワーク採用を再検討する
