# ADR-0001: 設計判断をADRとして記録する

- **日付**: 2026-07-06
- **ステータス**: 採用
- **関連**: `AGENTS.md`, `docs/adr/TEMPLATE.md`

## 背景（Context）

AIコーディングエージェントを併用する開発では、設計判断の「理由」がAIとの会話履歴の中に埋もれ、セッション終了とともに失われる。別のAIや未来の自分が「なぜこうなっているのか」を知る手段がないと、同じ議論の繰り返しや、意図を壊す変更が起きる。

## 検討した選択肢

### 選択肢A: ADR（Architecture Decision Record）を軽量Markdownで残す

- 長所: ツール不要・Git履歴と一体・どのAIも読める
- 短所: 書く習慣づけが必要

### 選択肢B: 会話ログやWiki等の外部ツールに残す

- 長所: 書くコストが低い（ログは自動で残る）
- 短所: リポジトリ外に情報が分散し、SSOT原則に反する。AIツールを変えるとアクセスできない

## 決定（Decision）

選択肢Aを採用する。重要な設計判断（ライブラリ選定・データ構造・アーキテクチャ・規約の大変更）は必ず `docs/adr/` に `TEMPLATE.md` 形式で記録する。

## 影響（Consequences）

- 良くなること: AI・人間を問わず、判断の経緯を30分以内に把握できる。判断の蒸し返しがなくなる
- 引き受けるコスト: 判断のたびに5〜10分の記述コスト
- 覆す条件: ADRが形骸化して書かれなくなった場合は、より軽量な形式（コミットメッセージへの集約など）を再検討する
