# レビュー手順

個人開発＋AI協働を前提に、「セルフレビュー」と「AIクロスレビュー」の2段構えとする。

## 1. セルフレビュー（コミット前・必須）

コミットする前に、diff全体を見ながら以下のチェックリストを通す。AI自身が実装した場合も、AIがこのチェックリストで自己点検する。

### チェックリスト

- [ ] **仕様との一致**: 該当アプリの `specs/` の仕様書どおりか。仕様にない挙動を勝手に追加していないか
- [ ] **完成の定義**: 仕様書の「完成の定義」をすべて満たしているか
- [ ] **動作確認**: 変更箇所を実際に動かした（またはテストを実行した）か
- [ ] **規約準拠**: `docs/coding-standards.md` に沿っているか。既存コードのスタイルと揃っているか
- [ ] **不要物の混入**: デバッグ用コード・console.log・コメントアウトの死体・無関係の変更が混ざっていないか
- [ ] **秘密情報**: APIキー・トークン・個人情報が含まれていないか
- [ ] **エラー処理**: 異常系（空入力・通信失敗など）で壊れないか
- [ ] **ドキュメント更新**: 仕様変更なら該当アプリの `specs/`、判断があれば `docs/adr/`、ユーザー向け変更なら該当アプリの `CHANGELOG.md`（基盤変更ならルートの `CHANGELOG.md`）を更新したか

## 2. AIクロスレビュー（任意・推奨）

大きめの変更では、**実装したAIとは別のAI（または別セッション）** にレビューさせると効果的。手順:

1. `prompts/review-code.md` のプロンプトを使う
2. レビュー対象は「ブランチのdiff」と「対応する仕様書」のセットで渡す
3. 指摘は鵜呑みにせず、仕様と規約に照らして採否を判断する
4. 採用した指摘で重要なものは、タスクファイルの作業ログに記録する

## 3. マージ基準

以下をすべて満たしたらマージできる。

1. セルフレビューのチェックリストが全項目クリア
2. テストが通っている（`docs/testing-policy.md`）
3. 該当する `CHANGELOG.md` が更新されている（ユーザーに見える変更の場合）
4. コミット履歴が `docs/git-workflow.md` に準拠している

## レビュー指摘の扱い

- 指摘には「must（直すべき）」「should（直したほうがよい）」「nit（好みの範疇）」の温度を付ける。
- must が残ったままマージしない。should/nit は対応をタスク化して後回しにしてよい。
