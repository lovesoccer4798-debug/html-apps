# レビュー手順

個人開発＋AI協働を前提に、「セルフレビュー」と「AIクロスレビュー」の2段構えとする。

## 1. セルフレビュー（コミット前・必須）

コミットする前に、diff全体を見ながら以下のチェックリストを通す。AI自身が実装した場合も、AIがこのチェックリストで自己点検する。

### チェックリスト

- [ ] **仕様との一致**: 該当アプリの `specs/` の仕様書どおりか。仕様にない挙動を勝手に追加していないか
- [ ] **完成の定義**: 仕様書の「完成の定義」をすべて満たしているか
- [ ] **動作確認**: 変更箇所を実際に動かした（またはテストを実行した）か
- [ ] **規約準拠**: `docs/coding-standards.md` に沿っているか。既存コードのスタイルと揃っているか
- [ ] **不要物の混入**: デバッグ用コード・console.log・コメントアウトの死体・無関係の変更が混ざっていないか
- [ ] **秘密情報**: APIキー・トークン・個人情報が含まれていないか
- [ ] **エラー処理**: 異常系（空入力・通信失敗など）で壊れないか
- [ ] **ドキュメント更新**: 仕様変更なら該当アプリの `specs/`、判断があれば `docs/adr/`、ユーザー向け変更なら該当アプリの `CHANGELOG.md`（基盤変更ならルートの `CHANGELOG.md`）を更新したか
- [ ] **Handbookの鮮度**: Phaseの完了を含む変更なら、Handbook表紙の「Today」と「品質パネル」を更新したか（それ以外の変更では触らなくてよい — [ADR](adr/20260714-handbook-living-map.md)）

## 2. AIクロスレビュー（任意・推奨）

大きめの変更では、**実装したAIとは別のAI（または別セッション）** にレビューさせると効果的。手順:

1. `prompts/review-code.md` のプロンプトを使う
2. レビュー対象は「ブランチのdiff」と「対応する仕様書」のセットで渡す
3. 指摘は鵜呑みにせず、仕様と規約に照らして採否を判断する
4. 採用した指摘で重要なものは、タスクファイルの作業ログに記録する

## 3. 初心者テスト（体験層の成果物に必須）

**体験層の成果物**（Portal・START_HEREなど「人を迎えるもの」）は、リリース前に初心者テストを行う。Brand Book六箇条「初心者を置いていかない」を検証プロセスとして実装したものである。

1. NESTを初めて触る人に**15分自由に使ってもらい、口出しせずに観察する**（隣でも画面共有でも）
2. 詰まった場所・迷った場所・表情が曇った瞬間をメモする
3. 結果を `docs/reviews/` のPhaseレビュー（または `newcomer-test-YYYYMMDD.md`）に記録し、直せるものはその場で直すか、タスク化する
4. 本物の初心者が確保できない間は、**文脈を持たない初見視点でのシミュレーション**（新しいAIセッション等）で代替してよい。ただし記録に**シミュレーションである旨を必ず明記**する（嘘をつかない）

## 4. マージ基準

以下をすべて満たしたらマージできる。

1. セルフレビューのチェックリストが全項目クリア
2. テストが通っている（`docs/testing-policy.md`）
3. 該当する `CHANGELOG.md` が更新されている（ユーザーに見える変更の場合）
4. コミット履歴が `docs/git-workflow.md` に準拠している

## 5. 引き算のレビュー（棚卸しの視点）

レビューは「足すべき設計」を探すだけでなく、**「もう要らないもの」を探す視点**も持つ。年数を重ねても軽く保てるDeveloper OSであり続けるため（Design Principles 原則7・Brand Book「育てる＝剪定も成長」）。これは**新しいガバナンスではなく、既存レビューに足すレンズ（問いの束）**である。専用文書・専用手続き・固定の周期は作らない。

**いつ通すか**: Phase完了レビューのたびに軽く1回。加えて、ルールや文書が「増えた」と感じたときに棚卸しパスとして。

**5つの問い**:

- 削除できるルールはないか（守られていない／意味を失ったルール）
- 重複している文書はないか（同じことを2箇所で管理していないか）
- 統合・整理できるADRはないか
- Futureへ送るべきもの（今すぐには不要になったもの）はないか
- 現在の思想で不要になったものはないか

**引き算の作法（壊さないために）**:

- **ADRは消さない。** 役目を終えたら「廃止／置換」ステータスに変える（履歴を残す＝誠実）
- 文書を統合するときは**リンク切れを残さない**（CIが番人）
- **ルールの削除・文書の統合など影響の大きい引き算は「提案」**として出し、オーナー承認を得てから実施する（分類ガバナンスと同じ）。誤字・死んだリンク・明らかに古い一文の除去は自律でよい
- **迷ったら消さない。** ただし「なぜ残すか」を一度は問う

## レビュー指摘の扱い

- 指摘には「must（直すべき）」「should（直したほうがよい）」「nit（好みの範疇）」の温度を付ける。
- must が残ったままマージしない。should/nit は対応をタスク化して後回しにしてよい。
