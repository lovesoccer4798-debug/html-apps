# 仕様: Task Calendar v6 — マイカレンダー（分類・表示切替）と共有への道

- **作成日**: 2026-07-15
- **最終更新**: 2026-07-15
- **ステータス**: 第1段階 実装済み（2026-07-15 オーナー承認: 上限8・擬似チップ採用・チップ全ビュー表示）
- **関連**: v4仕様（ルーティン）/ タスク `tasks/in-progress/20260714-task-calendar.md`

## 1. 目的

TimeTreeのように予定・タスクを「カレンダー」単位で分類し、見たいものだけ表示できるようにする（第1段階・ローカルのみ・無料）。将来はカレンダー単位の共有とログイン（第2段階・Firebase無料枠）につなげる。

## 2. スコープ — 第1段階（ローカル・サーバー不要）

### やること

- **マイカレンダーの作成・編集・削除**: ユーザーが自由に追加できる（例は「プライベート」「習慣」「仕事」だが固定しない）。名前＋色を設定
  - **上限は8個**（推奨・根拠は§7）。初期状態は「マイカレンダー」1個のみ
  - 削除時: そのカレンダーの項目は「マイカレンダー」へ移動（データは消さない）＋元に戻すトースト
- **項目の所属**: タスク・予定の追加／編集シートに「カレンダー」選択を追加（既定=マイカレンダー）。既存データは自動的にマイカレンダー所属
- **表示フィルタ（TimeTree風チップ）**: カレンダー画面（日／週／月／年）の上部にチップ列 — 「すべて」＋各カレンダー（チェック式）。チェックしたものだけ表示、「すべて」で全表示。選択状態は保存
- **色の優先順位**: 項目の個別色 ＞ ルーティン色 ＞ カレンダー色（予定表モードのラベル・色タグに適用）
- **ルーティンとの関係**: ルーティンのタスクは従来どおり（ルーティン色）。フィルタには「ルーティン」という擬似チップを用意し、習慣だけ隠す/見ることもできる
- **管理画面**: 設定に「マイカレンダー」セクション（追加・名前変更・色変更・削除・並び順）

### やらないこと（第1段階では）

- 共有・ログイン・複数端末同期（第2段階）
- カレンダーごとの通知・アイコン画像

## 3. データ（追加のみ・後方互換）

```json
{
  "calendars": [ { "id": "c-default", "name": "マイカレンダー", "color": "green", "order": 0 } ],
  "settings": { "calendarFilter": "all" }
}
```
- タスク・予定に `calendarId`（なし=c-default扱い）。`calendarFilter` は `"all"` または 表示中IDの配列

## 4. 第2段階の方向性（Firebase — 別仕様で詳細化）

- **Firebase Sparkプラン（無料枠）**: カード登録不要＝**自動課金が構造的に発生しない**（上限到達時は一時停止するだけ）。個人〜2人利用なら無料枠（Firestore 5万読取/日等）で十分
- ログイン: Firebase Auth（Googleログイン推奨・実装経験あり）。共有: カレンダー単位に招待コードで参加（TimeTreeと同じ粒度）。彼女と相互の予定見える化が最初のユースケース
- 技術メモ: 本リポジトリはCDN禁止のため、Firebase SDKは**リポジトリに同梱（自己ホスト）**する方針でADRを書く。オフライン時はローカル保存→復帰時に同期（衝突は後勝ち）
- 開始条件: 第1段階が安定して使えていること＋オーナーがFirebaseプロジェクト（無料）を用意

## 5. 完成の定義（第1段階）

- [ ] カレンダーを追加・名前変更・色変更・削除（項目はデフォルトへ移動）でき、上限で優しく制止される
- [ ] 追加／編集シートでカレンダーを選べ、チップで表示を絞れる（すべて/個別、選択は保存）
- [ ] 予定表モード・色タグにカレンダー色が反映（個別色・ルーティン色が優先）
- [ ] 既存データ・全回帰テストが無傷（追加フィールドのみ）

## 6. エッジケース

- 最後の1個は削除不可（既定カレンダーは常に存在）
- フィルタで全部外した場合は「すべて」に自動復帰（真っ白画面を作らない）
- 上限到達時: 追加ボタンを無効化し理由を表示

## 8. 将来メモ（オーナー構想・把握用）

- 無料範囲でのAPI連携（Googleカレンダー・Notion等）をビジョンボード後に検討。**アプリの設定画面から連携ON/OFFできる**構成にする。GoogleカレンダーAPIは個人利用無料、NotionAPIも無料枠あり。認証情報の置き場所（サーバーレスでの安全なOAuth）が主な設計課題

## 7. 論点（オーナー確認）

1. **上限は8個を推奨**。根拠: ①色パレットが6色＋既定で、8個を超えると色の見分けが実用的に破綻する ②チップ列が2行を超えると毎回の切替が億劫になり「分類疲れ」で使わなくなる（TimeTreeでも実利用は3〜5個が多い）③少なすぎる制限（3個）は「試しに分ける」余地を奪う。**8=自由さと見通しの折衷案**。3〜5個運用を想定しつつ天井は8、という考え方
2. ルーティン擬似チップ（習慣だけ隠せる）は入れてよいか？（推奨: 入れる — 仕事モードの時に習慣を非表示にできる）
3. チップ列は日／週／月／年すべてに出すか、月だけか？（推奨: すべて — 「今日」タブは常に全表示）
